# C4 — Фактическая архитектура MVP прототипа Auto-Flashcards

## 1. Введение

Этот документ описывает фактическую архитектуру MVP прототипа Auto-Flashcards, реализованного в рамках Лаборатории 3. Архитектура основывается на исходных схемах из Лабы 2, но отражает реальные решения, принятые во время разработки.

## 2. Отличия от планируемой архитектуры

- Фокус сделан на реализации end-to-end сценария для UC-0, UC-1 и UC-3 в упрощённом виде;
- Обработка сложных форматов документов перенесена в будущие спринты;
- Используется одна baseline LLM-модель без сложной оркестрации;
- Часть нефункциональных требований (масштабирование, продвинутый мониторинг) отложена;
- В качестве БД и системы аутентификации используется внешний сервис Supabase (PostgreSQL + Auth).

## 3. C4 Level 1 — Контекстная диаграмма (actual)

Участники:

- **Студент** — конечный пользователь;
- **Система Auto-Flashcards (Web-приложение)**;
- **Провайдер LLM** — внешняя AI-платформа или API;
- **Supabase** — внешний сервис (PostgreSQL + Auth);
- **Langfuse** — система трассировки вызовов LLM.

Краткое взаимодействие:

- Студент работает через браузер с веб-интерфейсом (Frontend, Next.js);
- Интерфейс обращается к Backend API (FastAPI);
- Backend использует:
  - Supabase PostgreSQL для хранения данных;
  - Supabase Auth для проверки токенов пользователей;
  - baseline AI-модель;
  - Langfuse для трассировки LLM-вызовов.

## 4. C4 Level 2 — Диаграмма контейнеров (actual)

Основные контейнеры:

1. **Frontend (Next.js)**

   - Расположение: `frontend/src/app/…`
   - Ответственность:
     - UI для регистрации/логина, создания колод и сессий повторения;
     - вызовы REST API backend;
     - отображение ответов AI и ошибок.

2. **Backend API (FastAPI)**

   - Расположение: `backend/app/main.py`, `backend/routers/…`
   - Ответственность:
     - REST API для:
       - регистрации/аутентификации (проверка токена Supabase);
       - работы с колодами и карточками;
       - работы с сессиями повторения;
       - обращения к AI-модели;
     - валидация запросов (Pydantic-модели);
     - логирование и базовая обработка ошибок.

3. **База данных (Supabase PostgreSQL)**

   - Доступна как часть внешнего сервиса Supabase;
   - Ответственность:
     - хранение информации о пользователях;
     - хранение колод и карточек;
     - хранение результатов сессий повторения.

4. **AI-модуль (Baseline модель)**

   - Расположение: `ml/models/baseline.py`;
   - Ответственность:
     - приём текста учебного материала;
     - генерация списка карточек (вопрос/ответ/доп. поля);
     - возврат результата backend-сервису.

5. **Langfuse**

   - Настраивается через `ml/tracing/langfuse_config.py` и `docker-compose.langfuse.yml`;
   - Ответственность:
     - трассировка вызовов LLM;
     - логирование промптов, ответов и ошибок.

6. **Инфраструктура и CI**

   - Docker-образы: `backend/Dockerfile`, `frontend/Dockerfile`;
   - Оркестрация: `docker-compose.yml`;
   - CI: `.github/workflows/ci.yml`.

7. **Supabase (внешний сервис: PostgreSQL + Auth)**
   - Ответственность:
     - управление пользователями и аутентификация (Supabase Auth);
     - предоставление управляемой PostgreSQL базы данных;
     - взаимодействие с backend через HTTP / SDK и токены доступа.

## 5. C4 Level 3 — Компоненты Backend (обзор)

Основные компоненты:

- `app/main.py`

  - точка входа приложения FastAPI;
  - подключение роутеров и middleware;
  - настройка CORS.

- `backend/routers/ai.py`

  - эндпоинты для генерации карточек;
  - интеграция с baseline AI-моделью;
  - вызовы Langfuse для трассировки.

- `backend/routers/decks.py` (или аналогичный модуль)

  - CRUD-операции для колод и карточек;
  - привязка колод к пользователю по идентификатору.

- `backend/routers/review.py`

  - эндпоинты для управления сессиями повторения;
  - сохранение результатов в БД.

- `backend/routers/auth.py` (если используется)

  - интеграция с Supabase Auth;
  - проверка токенов и получение информации о пользователе.

- `backend/models/…`
  - ORM-модели сущностей (пользователь, колода, карточка, результат сессии).

## 6. ADR — ключевые архитектурные решения

- **ADR-1: Стек FastAPI + Next.js**

  - выбран для быстрого прототипирования и чёткого разделения фронтенда и бэкенда.

- **ADR-2: Использование внешней LLM как baseline-модели**

  - позволяет быстро получить рабочий AI-функционал без развёртывания тяжёлых локальных моделей.

- **ADR-3: Внедрение Langfuse**

  - даёт возможность анализировать качество промптов и ответов, собирать метрики (latency, tokens и т.д.).

- **ADR-4: Использование Supabase (PostgreSQL + Auth)**
  - позволяет не реализовывать собственную систему аутентификации и управления базой данных;
  - снижает операционные затраты на настройку и поддержку БД и auth.

## 7. Интеграция с Supabase (Auth + Database)

В текущем MVP Supabase используется как внешний управляемый сервис, который объединяет:

- **PostgreSQL базу данных**:
  - хранение информации о пользователях;
  - хранение ссылок на колоды и связанные с ними данные;
- **Supabase Auth**:
  - регистрация и вход пользователя по email;
  - выдача access-токенов, которые передаются на backend.

Типовой поток аутентификации:

1. Пользователь выполняет регистрацию или вход через Supabase (во frontend используется Supabase SDK).
2. При успешной аутентификации Supabase выдаёт токен.
3. При обращении к защищённым эндпоинтам backend (создание колоды, запуск сессии повторения и т.п.) frontend отправляет этот токен в заголовках.
4. Backend проверяет токен через Supabase / библиотеку аутентификации и, в случае успеха, выполняет запрос от имени конкретного пользователя.

Такой подход позволяет:

- не реализовывать свою систему аутентификации с нуля;
- использовать управляемую PostgreSQL базу данных;
- упростить управление пользователями и их данными в прототипе.

## 8. End-to-End поток

1. Пользователь открывает веб-интерфейс (Frontend).
2. Выполняет регистрацию или вход через Supabase Auth.
3. На странице создания колоды вводит текст учебного материала и отправляет запрос.
4. Frontend вызывает endpoint Backend API, передавая токен аутентификации.
5. Backend:
   - валидирует входные данные;
   - проверяет токен пользователя (Supabase Auth);
   - вызывает baseline AI-модель;
   - логирует вызов через Langfuse;
   - сохраняет сгенерированные карточки в Supabase PostgreSQL.
6. Backend возвращает результат на Frontend.
7. Пользователь видит созданную колоду и может запустить сессию повторения.
8. Во время сессии:
   - Frontend запрашивает карточки у Backend;
   - пользователь оценивает каждую карточку;
   - результаты сохраняются в БД и привязываются к пользователю.

## 9. Ограничения и технический долг

- Алгоритм SRS упрощён, требует доработки и тестирования;
- Отсутствует полноценная ролевая модель и расширенная безопасность;
- Обработка больших и сложных учебных материалов отложена;
- Требуется дальнейший рефакторинг модулей backend и расширение тестов;
- Зависимость от внешних сервисов (Supabase, LLM-провайдер) создаёт риски при отсутствии сети.
