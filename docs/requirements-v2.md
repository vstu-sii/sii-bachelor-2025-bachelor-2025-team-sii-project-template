# Требования v2 — MVP прототип Auto-Flashcards

## 1. Введение

Данный документ описывает обновлённые требования к системе Auto-Flashcards на основе результатов Лаборатории 3 (MVP прототип).

Цели документа:

- зафиксировать, какая часть первоначальных требований из `docs/requirements.md` реализована в прототипе;
- описать отклонения от плана Лабы 2;
- уточнить приоритеты требований для следующего спринта.

## 2. Область действия MVP прототипа

MVP прототип реализует базовый сценарий работы системы:

- Пользователь (Студент) может:
  - зарегистрироваться / войти в систему (базовая аутентификация);
  - создать колоду (deck) из учебного материала на основе текста;
  - запустить простую сессию повторения карточек (spaced repetition);
  - просмотреть базовую статистику по колоде (количество карточек, количество выученных);
- Backend (FastAPI) предоставляет API для работы с колодами, карточками, сессиями и аутентификацией;
- Frontend (Next.js) отображает основные страницы и взаимодействует с API;
- База данных (через Supabase PostgreSQL) хранит пользователей, колоды, карточки и результаты повторения;
- Baseline AI-модель генерирует первоначальный набор карточек из текста;
- Langfuse используется для трассировки вызовов LLM.

Ограничения MVP:

- Поддерживается ограниченный размер входного текста;
- Обработка сложных форматов (большие PDF, сканы, мультимодальные материалы) пока отсутствует;
- Расширенная статистика и гибкие настройки профиля реализованы частично;
- Безопасность и управление ролями реализованы на базовом уровне.

## 3. Сопоставление с требованиями Лабы 2

### 3.1 Реализованные требования

- FR-0.x (Базовая аутентификация пользователя) — **реализовано частично**:
  - регистрация/логин через Supabase Auth;
  - проверка токена при обращении к защищённым эндпоинтам;
  - привязка данных (колод, сессий) к конкретному пользователю.
- FR-1.x (Создание колоды из учебного материала) — **реализовано частично**:
  - ввод текста / простого файла;
  - генерация набора карточек через LLM;
  - сохранение колоды в БД.
- FR-3.x (Ежедневная сессия повторения) — **реализовано частично**:
  - выбор карточек для сессии;
  - показ вопроса и ответа;
  - оценка карточки пользователем (например: Bad / Good / Easy);
  - сохранение результатов в БД.

### 3.2 Нереализованные / отложенные требования

- FR-2.x (Расширенное редактирование и управление колодами) — **реализовано минимально**:
  - базовое редактирование (изменение текста карточки, удаление);
  - отсутствует массовое редактирование, слияние дубликатов, история изменений.
- FR-4.x (Расширенная статистика и аналитика) — **отложено**:
  - графики по периодам, продвинутые фильтры и рекомендации пока не реализованы.
- FR-5.x (Полноценное управление профилем пользователя) — **отложено / минимально**:
  - используется упрощённый профиль;
  - настройки языка/темы/сложности в основном не реализованы.

### 3.3 Изменённые требования

В процессе разработки прототипа:

- Требования к поддержке больших файлов и сложных форматов перенесены в последующие спринты;
- Требования к производительности LLM-вызовов для MVP ослаблены (допускается повышенная задержка ответа);
- Добавлено новое требование:
  - FR-N: Все вызовы LLM должны логироваться в Langfuse для последующего анализа качества.

## 4. Валидация use-cases через прототип

### 4.0 UC-0: Регистрация и аутентификация пользователя

- Степень покрытия: **частично**
- Реализовано:
  - базовая аутентификация пользователя через Supabase Auth (email + пароль / magic link);
  - проверка токена аутентификации при обращении к защищённым эндпоинтам backend;
  - привязка созданных колод и результатов сессий повторения к конкретному пользователю.
- Ограничения:
  - нет полноценного управления ролями и правами доступа;
  - отсутствуют расширенные настройки безопасности (например, двухфакторная аутентификация);
  - часть функциональности (просмотр контента в режиме «гостя») может работать без строгой привязки к аккаунту.

### 4.1 UC-1: Создание колоды из учебного материала

- Степень покрытия: **частично**
- Работает:
  - ввод текста;
  - генерация карточек baseline-моделью;
  - сохранение колоды.
- Ограничения:
  - нет поддержки сложных файлов и большого объёма материала;
  - ограниченные возможности по постобработке карточек.

### 4.2 UC-2: Редактирование и улучшение колоды

- Степень покрытия: **минимально**
- Работает:
  - базовое редактирование текста карточек;
  - удаление карточек.
- Ограничения:
  - нет массового редактирования;
  - нет истории версий и отмены действий.

### 4.3 UC-3: Ежедневная сессия повторения (SRS)

- Степень покрытия: **частично**
- Работает:
  - выбор набора карточек для сессии;
  - показ вопроса и ответа;
  - сохранение оценки пользователя и результатов.
- Ограничения:
  - алгоритм SRS реализован в упрощённом виде;
  - нет гибких пользовательских настроек расписания.

### 4.4 UC-4: Просмотр статистики и прогресса

- Степень покрытия: **минимально**
- Работает:
  - отображение базовой статистики по колоде.
- Ограничения:
  - нет детализированных графиков и фильтров.

### 4.5 UC-5: Управление профилем пользователя

- Степень покрытия: **минимально**
- Работает:
  - упрощённый сценарий идентификации пользователя.
- Ограничения:
  - настройки профиля практически отсутствуют.

## 5. Отклонения от плана и причины

- Сложная обработка документов и расширенная статистика перенесены в будущие спринты из-за ограничений по времени;
- Алгоритм SRS реализован в базовом виде без глубокой настройки;
- UI сфокусирован на демонстрации основного end-to-end сценария.

## 6. Открытые вопросы и риски

- Как измерять и улучшать качество генерации карточек AI-моделью;
- Как масштабировать обработку больших объёмов учебного материала;
- Требования к безопасности и аутентификации пользователей (усиление Supabase-конфигурации, политики паролей и т.п.).
