# Отчёт по интеграционному тестированию MVP прототипа Auto-Flashcards

## 1. Введение

Цель данного отчёта — зафиксировать результаты end-to-end тестирования MVP прототипа Auto-Flashcards, созданного в рамках Лаборатории 3, и оценить готовность системы к демонстрации.

## 2. Методика интеграционного тестирования

- Все сервисы запускались командой `docker-compose up`:
  - frontend (Next.js),
  - backend (FastAPI),
  - Langfuse.
- В качестве базы данных и системы аутентификации использовался внешний сервис Supabase (PostgreSQL + Auth).
- Тестирование проводилось вручную по ключевым use-cases:
  - UC-0: регистрация и вход пользователя;
  - UC-1: создание колоды из учебного материала;
  - UC-3: сессия повторения (spaced repetition);
  - частично UC-2 и UC-4.
- Проверялись:
  - доступность фронтенда по адресу `http://localhost:3000`;
  - корректная работа API backend;
  - сохранение данных в Supabase PostgreSQL;
  - успешные вызовы AI-модели;
  - появление трасс в Langfuse.

## 3. Сценарии end-to-end тестов

### 3.1 Сценарий 1 — Регистрация и вход пользователя (UC-0)

**Шаги:**

1. Открыть веб-интерфейс по адресу `http://localhost:3000`.
2. Перейти на страницу регистрации / входа.
3. Выполнить регистрацию нового пользователя через Supabase (email + пароль / magic link).
4. Выполнить вход под созданным пользователем.
5. Перейти на главную страницу приложения и убедиться, что пользователь считается авторизованным.

**Ожидаемый результат:**

- регистрация и вход проходят без ошибок;
- токен аутентификации сохраняется на фронтенде;
- защищённые действия (создание колоды, запуск сессии) доступны только авторизованному пользователю.

**Фактический результат:**

- регистрация и вход успешно выполняются;
- после обновления страницы состояние авторизации сохраняется (при корректной настройке);
- защищённые эндпоинты backend доступны только при наличии валидного токена.

**Статус:** Успешно.

---

### 3.2 Сценарий 2 — Создание колоды и генерация карточек (UC-1)

**Шаги:**

1. Выполнить вход в систему (см. сценарий 1).
2. Перейти на страницу создания новой колоды.
3. Ввести текст учебного материала.
4. Нажать кнопку генерации колоды.
5. Дождаться ответа от системы.
6. Проверить отображение сгенерированных карточек и наличие записей в Supabase БД.

**Ожидаемый результат:**

- колода создаётся;
- карточки успешно генерируются AI-моделью;
- данные сохраняются в БД;
- в Langfuse фиксируется вызов LLM.

**Фактический результат:**

- колода создаётся и отображается во фронтенде;
- карточки генерируются, но при большом объёме текста возможны задержки ответа;
- записи о колоде и карточках присутствуют в БД;
- в Langfuse видны трассы LLM-вызовов.

**Статус:** Успешно, с небольшими ограничениями по производительности.

---

### 3.3 Сценарий 3 — Сессия повторения (UC-3)

**Шаги:**

1. Войти в систему под существующим пользователем.
2. Выбрать ранее созданную колоду.
3. Запустить сессию повторения.
4. Для каждой карточки:
   - просмотреть вопрос;
   - открыть ответ;
   - выбрать оценку (Bad / Good / Easy).
5. Завершить сессию и проверить сохранение результатов.

**Ожидаемый результат:**

- карточки показываются по очереди;
- оценки пользователя сохраняются;
- сессия завершается без ошибок.

**Фактический результат:**

- сессия запускается и карточки отображаются корректно;
- оценки отправляются на backend и сохраняются в БД;
- после завершения сессии ошибки не наблюдались.

**Статус:** Успешно.

---

### 3.4 Сценарий 4 — Просмотр базовой статистики (UC-4, частично)

**Шаги:**

1. Войти в систему.
2. Перейти на страницу статистики/прогресса.
3. Выбрать колоду.
4. Просмотреть базовые показатели (количество карточек, количество выученных и т.п.).

**Ожидаемый результат:**

- статистика рассчитывается и отображается корректно.

**Фактический результат:**

- базовые показатели (количество карточек, выученных и т.п.) отображаются;
- детальные графики и фильтры по периодам отсутствуют (не реализованы в MVP).

**Статус:** Частично успешно.

## 5. Найденные проблемы

- **BUG-1:** Задержка ответа AI-модели при большом объёме входного текста.

  - Влияние: замедляет UX при создании колоды.
  - Статус: требуется оптимизация или ограничение длины текста.

- **BUG-2:** Недостаточно информативные сообщения об ошибках на фронтенде.

  - Влияние: пользователю сложно понять, что именно пошло не так.
  - Статус: рекомендуется улучшить обработку ошибок в UI.

- **BUG-3:** Отсутствие расширенных возможностей статистики и фильтрации.
  - Влияние: ограничивает понимание прогресса пользователя.
  - Статус: запланировано к доработке в Спринте 4.

## 6. Выводы и рекомендации

- Основной end-to-end сценарий «регистрация → создание колоды → генерация карточек → сессия повторения» работает;
- Ограничения MVP связаны с:
  - производительностью AI-вызовов на больших текстах;
  - отсутствием расширенной статистики и удобного редактирования;
  - базовым уровнем UX и обработки ошибок.
- Рекомендуется:
  - оптимизировать работу с длинным текстом (ограничения, предобработка);
  - улучшить UX и сообщения об ошибках;
  - расширить покрытие автоматизированными тестами и включить их в CI;
  - развивать функциональность статистики и профиля пользователя в следующих спринтах.

Этот отчёт служит входными данными для планирования задач, описанных в `docs/sprint4-plan.md`.
