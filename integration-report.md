# Отчёт по интеграционному тестированию MVP прототипа Auto-Flashcards

## 1. Введение

Цель данного отчёта — зафиксировать результаты end-to-end тестирования MVP прототипа Auto-Flashcards, созданного в рамках Лабораторной 3, и оценить готовность системы к демонстрации.

В текущей версии реализован минимальный рабочий прототип:

- backend на FastAPI (`backend/app/main.py`, `backend/app/routers/ai.py`, `backend/app/routers/decks.py`);
- frontend на Next.js (`frontend/src/app/page.jsx`);
- baseline AI-модель генерации карточек (`ml/models/baseline.py`);
- заглушка для трассировки Langfuse (`ml/tracing/langfuse_config.py`);
- простое управление колодами в памяти процесса (без реальной БД).

Функциональность аутентификации (UC-0), полноценной сессии повторения (UC-3) и расширенной статистики (UC-4) пока **не реализована в коде**, описана только на уровне требований и архитектуры.

## 2. Методика интеграционного тестирования

Интеграционное тестирование проводилось в двух режимах:

1. **Локальный запуск без Docker:**
   - backend:
     ```bash
     py -m uvicorn backend.app.main:app --reload
     ```
     (запуск из корня репозитория, чтобы были доступны модули `backend` и `ml`);
   - frontend:
     ```bash
     cd frontend
     npm run dev
     ```
   - проверка API через Swagger (`http://127.0.0.1:8000/docs`);
   - проверка UI через браузер (`http://localhost:3000`).

2. **Запуск в Docker:**
   - сборка и запуск образа backend:
     ```bash
     docker build -t auto-flashcards-backend -f backend/Dockerfile .
     docker run --rm -p 8000:8000 auto-flashcards-backend
     ```
   - сборка и запуск образа frontend:
     ```bash
     docker build -t auto-flashcards-frontend -f frontend/Dockerfile .
     docker run --rm -p 3000:3000 auto-flashcards-frontend
     ```
   - запускается тот же самый end-to-end сценарий через браузер.

В ходе тестирования проверялись:

- доступность frontend по адресу `http://localhost:3000`;
- корректная работа API backend:
  - `GET /health`,
  - `POST /ai/generate`,
  - `GET/POST/DELETE /decks/`;
- корректное взаимодействие frontend ↔ backend;
- базовая корректность работы baseline AI-модели.

Supabase, Langfuse и внешняя БД **на данном этапе не подключены**. Данные о колодах хранятся в памяти процесса (in-memory storage).

## 3. Сценарии end-to-end тестов

### 3.1 Сценарий 1 — Проверка работоспособности API

**Шаги:**

1. Запустить backend (локально или в Docker).
2. Открыть `http://127.0.0.1:8000/health`.
3. Открыть `http://127.0.0.1:8000/docs`.
4. Через Swagger выполнить:
   - `POST /ai/generate` с коротким учебным текстом;
   - `GET /decks/` (список колод, изначально пустой);
   - `POST /decks/` (создание тестовой колоды);
   - `GET /decks/` (проверить, что колода появилась);
   - `DELETE /decks/{id}` (удалить колоду и убедиться, что она исчезла из списка).

**Ожидаемый результат:**

- `GET /health` возвращает `{"status": "ok"}`;
- `POST /ai/generate` возвращает список карточек (не менее 1–2 элементов);
- операции с `/decks/` выполняются без ошибок (201/200/204).

**Фактический результат:**

- health-check и документация доступны;
- baseline AI-модель возвращает список карточек по заданному тексту;
- операции с колодами работают в рамках in-memory хранилища.

**Статус:** Успешно.

---

### 3.2 Сценарий 2 — Генерация карточек из учебного текста через UI (UC-1, MVP)

**Шаги:**

1. Запустить backend и frontend.
2. Открыть веб-интерфейс по адресу `http://localhost:3000`.
3. В разделе генерации карточек:
   - вставить учебный текст (2–4 предложения);
   - нажать кнопку «Сгенерировать карточки».
4. Дождаться ответа и проверить отображение карточек на странице.

**Ожидаемый результат:**

- запрос уходит с frontend на `POST /ai/generate`;
- backend вызывает функцию `generate_flashcards` из `ml/models/baseline.py`;
- сгенерированные карточки отображаются в блоке «Сгенерированные карточки».

**Фактический результат:**

- запросы отправляются на backend (`/ai/generate`);
- baseline-модель корректно делит текст на предложения и формирует Q/A;
- карточки отображаются на фронтенде.

**Ограничения:**

- нет реального LLM, используется простая baseline-логика;
- аутентификация пользователя и привязка колод к отдельному профилю на этом шаге отсутствуют.

**Статус:** Частично успешно (UC-1 реализован на уровне MVP без учёта пользователей и постоянного хранилища).

---

### 3.3 Сценарий 3 — Управление колодами через UI (UC-2, частично)

**Шаги:**

1. Открыть `http://localhost:3000` при запущенных backend и frontend.
2. В блоке «Управление колодами»:
   - создать новую колоду (задать название и описание);
   - убедиться, что она появилась в списке колод;
   - при необходимости удалить колоду и проверить, что она исчезла из списка.

**Ожидаемый результат:**

- создание колоды вызывает `POST /decks/`;
- список колод получает данные через `GET /decks/`;
- удаление работает через `DELETE /decks/{id}`;
- UI обновляется после операций.

**Фактический результат:**

- создание и удаление колод работает;
- список колод обновляется после операций;
- данные хранятся только в памяти процесса (после перезапуска backend данные пропадают).

**Статус:** Частично успешно (UC-2 реализован без привязки к пользователю и без постоянной БД).

---

### 3.4 Статус остальных use-cases (UC-0, UC-3, UC-4)

- **UC-0: Регистрация и вход пользователя**

  - На уровне требований и архитектуры предусмотрено использование Supabase (Auth + PostgreSQL).
  - В текущем MVP **аутентификация не реализована**:
    - нет экранов логина/регистрации;
    - backend не проверяет токены.
  - **Статус:** Не реализовано в коде (только в документации).

- **UC-3: Сессия повторения (spaced repetition)**

  - Логика проведения сессии, расчёта интервалов и сохранения результатов повторений пока отсутствует.
  - В MVP пользователь может только сгенерировать карточки и просмотреть их.
  - **Статус:** Не реализовано.

- **UC-4: Просмотр статистики/прогресса**

  - Страница с аналитикой, графиками, продвинутой статистикой не реализована.
  - В текущей версии можно визуально оценить только список колод и сгенерированные карточки.
  - **Статус:** Не реализовано.

## 5. Найденные проблемы и ограничения

1. **Отсутствие постоянного хранилища данных (БД).**

   - Влияние: все колоды хранятся в памяти процесса; при перезапуске backend данные теряются.
   - Рекомендация: подключить Supabase/PostgreSQL или другую БД и перенести CRUD-операции из in-memory в ORM.

2. **Отсутствие аутентификации и привязки данных к пользователю.**

   - Влияние: все действия выполняются «анонимным» пользователем, нет разделения данных по аккаунтам.
   - Рекомендация: реализовать UC-0 (аккаунты через Supabase), добавить middleware/guard на backend.

3. **Ограниченная логика AI-модели (baseline вместо реального LLM).**

   - Влияние: качество карточек ограничено простым разбиением текста на предложения.
   - Рекомендация: интегрировать внешнего LLM-провайдера (OpenAI / локальная модель) и логировать запросы через Langfuse.

4. **Отсутствие реальной интеграции с Langfuse.**

   - Влияние: невозможно анализировать качество и производительность LLM-вызовов в продвинутом виде.
   - Рекомендация: заменить заглушку в `ml/tracing/langfuse_config.py` на реальный клиент Langfuse и добавить `docker-compose.langfuse.yml`.

5. **Отсутствие UI/логики для UC-3 и UC-4.**

   - Влияние: пользователь не может полноценно повторять материал по алгоритму spaced repetition и не видит прогресс.
   - Рекомендация: реализовать основной сценарий сессии повторения и страницу статистики в следующих спринтах.

6. **Ограниченная обработка ошибок на фронтенде.**

   - Влияние: пользователю не всегда понятно, что произошло при сетевых ошибках или ошибках валидации.
   - Рекомендация: улучшить UX: отображать понятные сообщения об ошибках, состояния загрузки и пустые состояния.

---

## 6. Выводы и рекомендации

- MVP прототип Auto-Flashcards **технически работоспособен**:
  - пользователь может в браузере ввести учебный текст;
  - система генерирует базовые карточки через backend и baseline-модель;
  - есть минимальное управление колодами на уровне in-memory;
  - backend и frontend могут запускаться как локально, так и в Docker-контейнерах.

- Текущая версия соответствует целям Лабораторной 3:  
  продемонстрирована **техническая осуществимость** концепции и базовая интеграция компонентов (UI ↔ API ↔ AI-модель).

- Основные направления доработки на следующие спринты:
  - подключить постоянное хранилище (Supabase/PostgreSQL) и реализовать полноценные модели БД;
  - реализовать UC-0 (регистрация/вход), UC-3 (сессия повторения) и UC-4 (статистика);
  - заменить baseline-модель на реальный LLM-провайдер и завершить интеграцию с Langfuse;
  - улучшить UX/UI (обработка ошибок, loading-состояния, валидация форм);
  - добавить автоматизированные тесты и расширить CI-конвейер.

Данный отчёт служит входными данными для планирования задач, описанных в `docs/sprint4-plan.md`, и демонстрирует, что базовая архитектура и выбранный стек технологий жизнеспособны для развития проекта до полноценного продукта.
